# 账号绑定、按账号计费与换机付费迁移

## 摘要

本文档在现有 TOTP 客户端能力（见 \cite{SRS01}）之上，规定 **用户身份与支付侧** 的增量需求：支持将产品账户与 **微信、QQ、支付宝** 等国内常用渠道进行 **绑定**（用于登录识别与/或支付）；采用 **按账号收费** 的商业模式（**一个产品用户一笔费用**，见 §2.2）；用户 **更换手机** 时，通过 **付费** 获得受控的 **凭据/数据迁移** 或等效服务。本文档为需求层描述，不涉及具体 SDK 版本与接口字段；支付与开放平台对接须在 **IDD/合规评审** 后定稿。

## 关键词

第三方账号绑定；微信；QQ；支付宝；订阅与计费；按账号收费；换机迁移；TOTP

## 1. 范围与关系

- **在范围内**：账号体系与第三方绑定策略、计费维度（**按产品用户 / 第三方登录身份**，见 §2.2）、换机场景下的付费能力与交付物定义。
- **不在本文细化**：各平台开放能力清单、费率、发票、退款细则、服务端部署拓扑（见后续 **SAD/DDD**）。
- **与现有 SRS 关系**：不改变 \cite{SRS01} 中 TOTP 算法与本地 JSON 演示存储的 **技术事实**；商业化后 **密钥与备份** 的存储形态须单独 **安全需求** 评审。

## 2. 功能需求

### 2.1 第三方账号绑定

- 系统应支持用户将 **产品账户** 与以下至少一类渠道完成绑定：**微信**、**QQ**、**支付宝**（可扩展同类的「国内常用 OAuth / 联合登录」）。
- 绑定用途须在产品内明示，至少区分：**（A）身份与登录**、**（B）支付扣款**；若同一渠道兼用，须在隐私说明与授权流程中分项列出。
- 用户应能 **解绑** 或 **更换绑定**（受风控规则约束，具体规则待业务定稿）。
- **冲突处理**：同一第三方账号已被其他产品账户占用时，应拒绝重复绑定并给出可理解提示。

### 2.2 付费与计费模型

- 产品采用 **收费使用**；**计费单元 $$U$$** 定义为：**一个已通过第三方登录识别的产品用户**（业务口径：**以微信账号登录成功即视为一个计费用户**；QQ、支付宝等其它登录方式若开放，**各自**的登录身份与产品用户的一对一映射规则须一致且无重复计费歧义）。
- **不按**「每条 TOTP 密钥 / 每条条目」重复计费：同一计费用户下可保存 **多条** TOTP 条目，**不**因条目数增加而自动叠加基础席位费。若未来产品需设 **单用户条目上限** 或 **超限加价**，须在定价与订单模型中单独立项，不在 v1.0.1 默认假设内。
- 应支持 **预付费** 或 **周期订阅** 中至少一种（可并存），与微信/QQ/支付宝等 **支付通道** 对接完成下单、支付结果回调与订单查询。
- 未付费或权益不足时，应 **明确限制** 能力（例如禁止登录后新增条目、禁止导出/换机，具体与 §2.3 一致），且界面提示原因。

### 2.3 换机（迁移）付费

- 用户因 **更换手机** 需要在新设备上恢复 TOTP 数据时，应提供 **付费** 的 **换机/迁移** 流程。
- 迁移交付物至少满足其一（实现选型待定）：**加密备份包** 在用户授权下由旧设备生成并由新设备导入；或 **服务端托管** 的加密同步（须符合密钥不上明文存储等安全基线）。
- 换机费用可与 **按账号** 权益组合销售（例如含 $$N$$ 次换机），须在订单与权益模型中可追溯。

## 3. 非功能需求（摘录）

- **合规**：收集微信/QQ/支付宝用户标识、支付信息须符合各平台协议及《个人信息保护法》等；支付需具备相应资质或委托持牌机构。
- **安全**：绑定与支付令牌不得以明文写入客户端日志；迁移通道须防中间人篡改与重放（具体见后续安全设计）。
- **可用性**：绑定、支付、换机关键步骤须有明确状态与失败重试指引。

## 4. 待确认项（产品/法务）

**注**：原 C-01「按条数还是按用户计费」已于 **v1.0.1** 确认并写入 **§2.2**（按第三方登录用户计费）。

| 编号 | 事项 |
|------|------|
| C-02 | 换机是否 **必须** 依赖已绑定第三方账号以做身份校验。 |
| C-03 | 是否需 **发票**、**企业采购**、**退款** 策略。 |
| C-04 | 单用户下 TOTP 条目 **是否设上限**；若设，超限是否加价或仅禁止新增。 |
